# Перечень тех заданий

Перечень технических заданий по проекту «АвтоСнаб» (MVP)

Настоящий документ содержит технические задания для участников команды разработки MVP продукта «АвтоСнаб».

Каждое техническое задание определяет цели, область ответственности, состав выполняемых работ, требования к результатам, ограничения и критерии приемки.

Перечень технических заданий отражает функциональные роли, необходимые для разработки MVP продукта, и не определяет организационную структуру команды.

В зависимости от состава команды, компетенций специалистов и принятой модели организации работ одна роль может выполняться одним специалистом, совмещаться одним специалистом с другими ролями либо распределяться между несколькими специалистами.

При распределении роли между несколькими специалистами зоны ответственности, состав выполняемых работ и ожидаемые результаты должны быть определены таким образом, чтобы обеспечить выполнение требований соответствующего технического задания в полном объеме и исключить дублирование функций или возникновение нераспределенных зон ответственности.

В рамках проекта разрабатываются следующие технические задания.

**1\. Бизнес-аналитик**

**Техническое задание:** анализ и формализация бизнес-требований, описание бизнес-процессов и бизнес-логики, разработка и актуализация аналитической и проектной документации в части бизнес-требований, постановка задач команде разработки в части реализации бизнес-требований и сопровождение их реализации.

**2\. Системный архитектор**

**Техническое задание:** проектирование архитектуры системы, структуры модулей, потоков данных, интерфейсов взаимодействия, архитектурных ограничений и контроль соответствия реализации утвержденной архитектуре.

**3\. Backend-разработчик**

**Техническое задание:** разработка серверной части системы, реализация бизнес-логики обработки данных, взаимодействия модулей, автоматических расчетов, контрольной проверки документов и подготовки данных к экспорту.

**4\. Frontend-разработчик**

**Техническое задание:** разработка пользовательского интерфейса, экрана контроля качества данных, механизмов отображения результатов обработки, статусов, ошибок, предупреждений и взаимодействия пользователя с системой.

**5\. Специалист по OCR**

**Техническое задание:** разработка модуля распознавания товарных накладных, извлечение реквизитов документа и товарных позиций, подготовка исходных данных для последующей обработки.

**6\. AI Engineer / LLM Engineer**

**Техническое задание:** разработка интеллектуального модуля обработки данных, нормализация наименований, анализ фасовок, определение коэффициентов пересчета, обработка нестандартных случаев и передача результатов Backend.

**7\. Инженер по интеграции**

**Техническое задание:** разработка механизмов интеграции с учетной системой, получение справочных данных, экспорт обработанных документов, контроль обмена данными и обработка ошибок интеграции.

**8\. QA Engineer (инженер по тестированию)**

**Техническое задание:** разработка тестовой документации, проведение функционального, интеграционного и приемочного тестирования, регистрация, анализ и контроль устранения выявленных дефектов.

**9\. DevOps / инженер сопровождения**

**Техническое задание:** разработка и сопровождение инфраструктуры проекта, развертывание рабочих сред, обеспечение стабильности работы системы, мониторинг, журналирование, резервное копирование и техническое сопровождение MVP.

**10\. Руководитель проекта**

**Техническое задание:** управление проектом разработки MVP, планирование этапов реализации, координация участников, контроль сроков, управление изменениями, рисками и организацией взаимодействия команды.  
**Техническое задание**

 Бизнес-аналитик  
**Проект: MVP продукта «АвтоСнаб»**

1\. Назначение  
Обеспечить формализацию бизнес-требований, описание бизнес-процессов, сопровождение разработки и контроль соответствия реализуемой системы утвержденным бизнес-требованиям MVP продукта «АвтоСнаб».  
Бизнес-аналитик обеспечивает формализацию, актуализацию и единое понимание бизнес-требований всеми участниками проекта.  
 

2\. Основные задачи  
Обеспечить:

* сбор и анализ требований;  
* описание бизнес-процессов;  
* формализацию бизнес-логики;  
* постановку задач разработчикам;  
* согласование решений с заказчиком;  
* сопровождение разработки;  
* участие в приемке функционала.

 

3\. Анализ бизнес-процессов  
Необходимо описать существующий процесс обработки товарных накладных.  
В рамках анализа определить:

* участников процесса;  
* последовательность операций;  
* используемые документы;  
* точки принятия решений;  
* ручные операции;  
* проблемные участки;  
* ожидаемый результат автоматизации.

Все процессы должны быть описаны до начала разработки.  
 

4\. Формирование требований  
Необходимо подготовить и поддерживать в актуальном состоянии:

* функциональные требования;  
* бизнес-требования;  
* пользовательские сценарии;  
* ограничения MVP;  
* критерии приемки.

Все требования должны соответствовать архитектуре продукта.  
 

5\. Формализация бизнес-логики  
Необходимо описать правила:

* обработки товарных накладных;  
* сопоставления товаров;  
* сопоставления поставщиков;  
* определения фасовок;  
* определения коэффициентов пересчета;  
* выполнения контрольной проверки;  
* подготовки данных к экспорту.

Все правила должны быть согласованы до начала реализации.  
 

6\. Работа с архитектурной документацией  
Бизнес-аналитик обязан поддерживать актуальность следующих документов:

* Архитектура MVP продукта;  
* рабочие спецификации;  
* паспорт полей;  
* описание бизнес-процессов;  
* сценарии работы системы;  
* матрицы статусов;  
* матрицы ошибок;  
* технические задания.

При изменении бизнес-логики документация обновляется до начала разработки соответствующего функционала.  
 

7\. Постановка задач разработчикам  
Для каждого функционального блока необходимо подготовить отдельное техническое задание.  
Постановка задачи должна содержать:

* цель;  
* описание функциональности;  
* ограничения;  
* входные данные;  
* выходные данные;  
* бизнес-правила;  
* критерии приемки.

Технические задания должны быть достаточными для реализации без дополнительной интерпретации требований.  
 

8\. Сопровождение разработки  
В процессе реализации необходимо:

* отвечать на вопросы команды разработки;  
* уточнять бизнес-правила;  
* устранять противоречия в требованиях;  
* согласовывать изменения;  
* контролировать соответствие реализации архитектуре.

Изменения требований должны быть документированы.  
 

9\. Участие в тестировании  
Бизнес-аналитик принимает участие в:

* подготовке сценариев тестирования;  
* проверке реализованного функционала;  
* анализе выявленных дефектов;  
* приемке выполненных работ.

Приемка осуществляется на основании утвержденных требований и критериев приемки.  
 

10\. Ограничения  
Бизнес-аналитик не принимает технических решений по реализации системы.  
Все архитектурные и программные решения принимаются совместно с системным архитектором и командой разработки.  
Изменение утвержденной бизнес-логики без согласования с владельцем продукта не допускается.  
 

11\. Критерии приемки  
Работа бизнес-аналитика считается выполненной, если:

* бизнес-процессы полностью описаны;  
* требования согласованы;  
* архитектурная документация актуальна;  
* технические задания подготовлены;  
* разработчики обеспечены необходимой информацией;  
* реализованный функционал соответствует утвержденным требованиям.

 

12\. Результаты работы  
По завершении работ должны быть подготовлены:

* комплект бизнес-требований;  
* описание бизнес-процессов;  
* архитектурная документация;  
* комплект технических заданий;  
* пользовательские сценарии;  
* критерии приемки;  
* журнал изменений требований.

**Техническое задание**

Системный архитектор  
**Проект: MVP продукта «АвтоСнаб»**

1\. Назначение  
Обеспечить разработку, сопровождение и контроль архитектуры MVP продукта «АвтоСнаб», определить принципы взаимодействия всех модулей системы, обеспечить целостность архитектурных решений и возможность дальнейшего масштабирования продукта.  
Системный архитектор является владельцем архитектуры системы и отвечает за согласованность технических решений на протяжении всего жизненного цикла разработки.  
 

2\. Основные задачи  
Обеспечить:

* разработку архитектуры системы;  
* проектирование взаимодействия модулей;  
* определение потоков данных;  
* проектирование интеграций;  
* контроль архитектурной целостности;  
* сопровождение разработки;  
* анализ архитектурных изменений.

 

3\. Проектирование архитектуры  
Разработать архитектуру системы, включающую следующие модули:

* модуль загрузки документов;  
* модуль OCR;  
* модуль AI;  
* модуль сопоставления;  
* модуль анализа фасовок;  
* модуль проверки данных;  
* модуль пользовательских исправлений;  
* модуль контрольной проверки;  
* модуль интеграции с учетной системой.

Архитектура должна соответствовать утвержденному документу «Архитектура MVP продукта "АвтоСнаб"».  
 

4\. Проектирование потоков данных  
Определить порядок движения данных между всеми компонентами системы.  
Для каждого этапа необходимо определить:

* источник данных;  
* входные данные;  
* выполняемую обработку;  
* выходные данные;  
* следующий модуль обработки.

На протяжении всего процесса должна сохраняться целостность данных.  
 

5\. Проектирование взаимодействия модулей  
Для каждого модуля необходимо определить:

* область ответственности;  
* входные интерфейсы;  
* выходные интерфейсы;  
* условия запуска;  
* результаты работы;  
* правила обработки ошибок.

Модули должны быть максимально независимыми друг от друга.  
 

6\. Архитектурные ограничения  
При проектировании необходимо обеспечить следующие принципы:

* модульность;  
* масштабируемость;  
* расширяемость;  
* отказоустойчивость;  
* повторное использование компонентов;  
* отсутствие дублирования бизнес-логики.

Любые изменения архитектуры должны выполняться без нарушения существующих интерфейсов взаимодействия.  
 

7\. Контроль реализации  
В процессе разработки необходимо контролировать соответствие реализации утвержденной архитектуре.  
Проверке подлежат:

* структура модулей;  
* взаимодействие сервисов;  
* распределение ответственности;  
* отсутствие дублирования функциональности;  
* соответствие потоков данных архитектурной модели.

При выявлении отклонений архитектор определяет необходимые изменения.  
 

8\. Управление архитектурными изменениями  
Все предложения по изменению архитектуры должны проходить предварительный анализ.  
Для каждого изменения необходимо определить:

* цель изменения;  
* влияние на существующую архитектуру;  
* влияние на взаимодействие модулей;  
* влияние на интеграции;  
* влияние на производительность;  
* влияние на дальнейшее развитие продукта.

Изменения утверждаются до начала разработки.  
 

9\. Взаимодействие с командой проекта  
Системный архитектор взаимодействует с:

* бизнес-аналитиком;  
* Backend-разработчиком;  
* Frontend-разработчиком;  
* AI Engineer;  
* специалистом OCR;  
* инженером по интеграции;  
* QA Engineer;  
* DevOps;  
* руководителем проекта.

По каждому архитектурному вопросу архитектор обеспечивает единый технический подход для всех участников разработки.  
 

10\. Ограничения  
Системный архитектор не изменяет бизнес-требования самостоятельно.  
Все изменения бизнес-логики предварительно согласовываются с бизнес-аналитиком и владельцем продукта.  
Архитектурные решения не должны изменять утвержденную структуру данных MVP без официального пересмотра архитектурной документации.  
 

11\. Критерии приемки  
Работа системного архитектора считается выполненной, если:

* архитектура полностью описана;  
* взаимодействие всех модулей определено;  
* потоки данных формализованы;  
* зоны ответственности компонентов разграничены;  
* архитектура обеспечивает реализацию полного цикла обработки товарной накладной;  
* решения допускают дальнейшее развитие продукта без переработки базовой архитектуры.

 

12\. Результаты работы  
По завершении работ должны быть подготовлены:

* архитектурная схема системы;  
* описание модулей;  
* описание потоков данных;  
* описание интерфейсов взаимодействия;  
* архитектурные ограничения;  
* журнал архитектурных изменений;  
* рекомендации по развитию архитектуры продукта.

**Техническое задание**

Backend-разработчик  
**Проект: MVP продукта «АвтоСнаб»**

1\. Назначение  
Разработать серверную часть MVP продукта «АвтоСнаб», обеспечивающую полный цикл обработки товарной накладной — от получения результатов OCR до передачи проверенных данных в учетную систему.  
Backend является центральным модулем системы и отвечает за выполнение бизнес-логики, взаимодействие всех функциональных модулей, хранение промежуточных результатов обработки и формирование данных для пользовательского интерфейса.  
 

2\. Основные задачи  
Разработать серверную логику, обеспечивающую:

* прием результатов OCR;  
* передачу данных в AI-модуль;  
* сопоставление товаров и поставщиков;  
* автоматический анализ фасовок;  
* применение коэффициента пересчета и выполнение расчетов.;  
* пересчет количества и стоимости;  
* формирование данных для экрана проверки;  
* обработку пользовательских изменений;  
* контрольную проверку документа;  
* экспорт данных в учетную систему.

 

3\. Общая последовательность обработки  
Backend должен реализовать следующую последовательность:

1. Получение результатов OCR.  
2. Передача данных AI-модулю.  
3. Получение результатов AI.  
4. Сопоставление товаров со справочником.  
5. Сопоставление поставщика.  
6. Запуск анализа фасовок.  
7. Применение коэффициента пересчета и выполнение расчетов..  
8. Пересчет расчетных полей.  
9. Формирование данных экрана проверки.  
10. Получение пользовательских изменений.  
11. Повторный пересчет зависимых полей.  
12. Контрольная проверка документа.  
13. Подготовка данных к экспорту.  
14. Передача данных в учетную систему.  
15. Фиксация результата обработки.

 

4\. Реализуемые модули  
Backend обеспечивает работу следующих модулей:

* обработка документов;  
* взаимодействие с OCR;  
* взаимодействие с AI;  
* сопоставление товаров;  
* сопоставление поставщиков;  
* анализ фасовок;  
* расчет коэффициентов;  
* расчет стоимости;  
* обработка пользовательских изменений;  
* контрольная проверка;  
* экспорт данных.

 

5\. Работа с данными  
Backend использует только утвержденную структуру данных MVP.  
Запрещается:

* создавать дополнительные пользовательские поля;  
* изменять структуру утвержденного экрана проверки данных;  
* изменять назначение существующих полей без согласования.

Все операции должны выполняться исключительно с полями, утвержденными в листе «Накладная».  
 

6\. Работа со справочниками  
Backend должен обеспечить:

* поиск поставщика;  
* поиск товара;  
* получение данных из учетной системы;  
* получение учетной единицы измерения;  
* получение необходимых признаков товара;  
* работу со справочником исключений.

При отсутствии соответствия система должна передавать информацию в экран проверки данных.  
 

7\. Анализ фасовок  
Backend реализует алгоритм, утвержденный архитектурой продукта.  
Последовательность обработки:

1. получение исходной фасовки;  
2. определение единицы измерения;  
3. стандартный пересчет;  
4. проверка справочника исключений;  
5. обращение к AI при необходимости;  
6. расчет коэффициента пересчета;  
7. пересчет количества;

* итоговые значения документа;

Backend реализует функцию «Проверить документ».  
Во время проверки необходимо выполнить:

* «Есть предупреждения»;

Экспорт выполняется только для документов, имеющих разрешающий статус.  
После успешной передачи данных необходимо:

* зафиксировать результат;  
* изменить статус документа;  
* сохранить журнал операции.

 

11\. Логирование

* начало обработки;  
* завершение каждого этапа;  
* ошибки обработки;  
* предупреждения;  
* пользовательские изменения;  
* выполнение контрольной проверки;

12\. Требования к реализации  
Backend должен быть модульным.  
Каждый функциональный блок должен иметь четко определенную зону ответственности.  
Изменения одного модуля не должны требовать переработки остальных модулей.  
Все бизнес-правила должны быть сосредоточены в серверной логике и не должны дублироваться в пользовательском интерфейсе.

1\. Назначение  
Разработать модуль OCR, обеспечивающий максимально точное извлечение информации из товарных накладных независимо от поставщика, формата документа и качества исходного изображения.  
OCR является первым этапом обработки данных и отвечает исключительно за извлечение информации из документа без анализа бизнес-логики.  
 

2\. Основные задачи  
Разработать OCR-модуль, обеспечивающий:

* прием изображения или электронного документа;  
* распознавание текста;  
* выделение реквизитов накладной;  
* выделение товарных строк;  
* передачу результатов Backend без изменения исходных данных.

 

3\. Область ответственности  
OCR отвечает только за распознавание информации.  
OCR не выполняет:

* сопоставление товаров;  
* определение фасовок;  
* определение коэффициентов пересчета;  
* анализ единиц измерения;  
* вычисления;  
* исправление данных;  
* экспорт.

Любая последующая обработка выполняется другими модулями системы.  
 

4\. Поддерживаемые входные данные  
Модуль должен обеспечивать обработку:

* фотографий накладных;  
* отсканированных изображений;  
* PDF-документов;  
* электронных документов, представленных в виде изображений.

 

5\. Распознавание документа  
OCR должен определить:

* поставщика;  
* номер накладной;  
* дату накладной;  
* перечень товарных позиций;  
* итоговые суммы документа (при наличии);  
* прочие реквизиты, присутствующие в документе.

Каждая товарная строка должна быть выделена как самостоятельная запись.  
 

6\. Распознавание товарных строк  
Для каждой товарной позиции OCR должен максимально точно извлечь исходную информацию из документа.  
Необходимо распознавать текст без попыток его нормализации или интерпретации.  
Результат должен максимально соответствовать содержимому оригинальной накладной.  
 

7\. Требования к извлекаемым данным  
OCR должен сохранять:

* исходное написание товара;  
* исходные единицы измерения;  
* исходные числовые значения;  
* исходные обозначения фасовок;  
* исходные сокращения.

Исправление возможных ошибок распознавания выполняется на последующих этапах обработки.  
 

8\. Требования к качеству распознавания  
Модуль должен корректно работать при наличии:

* различных форм накладных;  
* различных поставщиков;  
* различных шрифтов;  
* различных вариантов расположения реквизитов;  
* фотографий среднего качества;  
* небольших перекосов изображения.

При невозможности достоверного распознавания отдельных элементов OCR должен вернуть исходный результат без искусственного исправления данных.  
 

9\. Предварительная обработка изображения  
При необходимости модуль должен выполнять:

* выравнивание изображения;  
* удаление шумов;  
* повышение контрастности;  
* улучшение читаемости текста;  
* подготовку изображения к распознаванию.

Все операции должны выполняться автоматически.  
 

10\. Передача результатов Backend  
После завершения распознавания OCR передает Backend:

* реквизиты документа;  
* список товарных строк;  
* исходный текст каждой строки;  
* координаты элементов документа (при наличии такой возможности);  
* информацию о качестве распознавания;  
* признаки участков с низкой уверенностью распознавания.

OCR не должен изменять структуру документа перед передачей результатов.  
 

11\. Обработка ошибок  
При невозможности полного распознавания документа OCR должен:

* передать все успешно распознанные данные;  
* отметить проблемные участки;  
* сохранить информацию о качестве распознавания;  
* не блокировать дальнейшую обработку документа.

Окончательное решение о возможности дальнейшей обработки принимается Backend.  
 

12\. Критерии приемки  
Работа OCR-модуля считается соответствующей требованиям, если:

* корректно выделяются реквизиты документа;  
* корректно выделяются товарные строки;  
* сохраняется исходное содержание документа;  
* результаты передаются Backend без потери информации;  
* OCR не выполняет функции, относящиеся к AI или бизнес-логике;  
* система обеспечивает возможность последующей обработки распознанных данных всеми остальными модулями архитектуры.

* обработки ошибок интеграции;  
* регистрации результатов передачи данных.

Интеграционный модуль должен обеспечивать получение данных, необходимых для обработки товарных накладных.  
Получаемые данные:

* справочник товаров;  
* справочник поставщиков;  
* учетные единицы измерения;  
* дополнительные признаки товаров, используемые при обработке документа.

Полученные данные должны использоваться Backend без изменения их структуры.  
 

4\. Передача данных в учетную систему  
Передача выполняется только после успешного завершения контрольной проверки документа.  
Интеграционный модуль получает от Backend финальный набор данных и обеспечивает его передачу в учетную систему.  
Интеграция не должна выполнять изменение или перерасчет данных.  
 

5\. Проверка технической готовности данных к передаче  
Перед отправкой необходимо проверить:

* полноту полученных данных;  
* соответствие обязательных полей;  
* корректность структуры передаваемого документа;  
* готовность документа к экспорту.

При обнаружении ошибок передача данных не выполняется.

После передачи данных необходимо определить результат выполнения операции.  
Для каждого документа фиксируется один из статусов:

* успешно передан;  
* передан с предупреждениями;  
* ошибка передачи;  
* отклонен учетной системой.

Полученный статус передается Backend для отображения пользователю.

Интеграционный модуль должен корректно обрабатывать:

* отсутствие соединения;  
* недоступность учетной системы;  
* ошибки авторизации;  
* ошибки формата данных;  
* ошибки API;  
* частичную передачу данных;  
* превышение времени ожидания ответа.

Ошибки должны регистрироваться без потери информации о документе.  
 

8\. Повторная передача  
При временных технических ошибках должна быть предусмотрена возможность повторной передачи документа.  
Повторная отправка допускается только для документов, не зарегистрированных в учетной системе как успешно обработанные.  
Повторная передача не должна приводить к созданию дублирующих документов.  
 

9\. Логирование  
Интеграционный модуль должен журналировать:

* дату и время передачи;  
* идентификатор документа;  
* результат передачи;  
* описание ошибки (при наличии);  
* ответ учетной системы;  
* информацию о повторных попытках передачи.

История обмена должна сохраняться для последующего анализа.  
 

10\. Требования к архитектуре  
Интеграционный модуль должен быть независим от внутренней реализации учетной системы.  
Все особенности конкретной учетной системы должны быть локализованы внутри интеграционного слоя.  
Изменение интерфейса взаимодействия с учетной системой не должно требовать изменения бизнес-логики Backend.  
 

11\. Ограничения  
Интеграционный модуль не должен:

* изменять данные документа;  
* выполнять сопоставление товаров;  
* выполнять пересчет фасовок;  
* изменять результаты AI-обработки;  
* выполнять контрольную проверку документа.

Все перечисленные операции выполняются другими компонентами системы.  
 

12\. Критерии приемки  
Работа интеграционного модуля считается соответствующей требованиям, если:

* справочники учетной системы успешно доступны Backend;  
* обработанные документы корректно передаются в учетную систему;  
* результаты передачи фиксируются системой;

# Архитектура обработки фасовок в MVP «АвтоСнаб»

Архитектура обработки фасовок в MVP «АвтоСнаб»

1\. Назначение

Архитектура обработки фасовок предназначена для автоматического определения фасовки товара, единицы измерения, коэффициента пересчета, количества в учетных единицах и цены в учетных единицах при обработке товарной накладной.

Основная цель — отказаться от создания большого справочника фасовок и заменить его последовательной логикой обработки данных.

2\. Общий принцип

Система не должна хранить все возможные варианты фасовок поставщиков в большом справочнике.

Фасовка определяется в момент обработки накладной на основании данных, полученных из документа, справочника товаров учетной системы, стандартных правил пересчета, справочника исключений и AI-анализа.

Общая последовательность обработки:

накладная  
 ↓  
 OCR  
 ↓  
 AI-структурирование  
 ↓  
 парсер фасовки  
 ↓  
 арифметический пересчет  
 ↓  
 справочник исключений  
 ↓  
 AI-анализ спорных случаев  
 ↓  
 ручное уточнение пользователем при необходимости  
 ↓  
 пересчет количества и цены в учетные единицы  
 ↓  
 экран контроля качества данных

3\. Входные данные

Для анализа фасовки используются следующие данные:

* наименование товара из накладной;  
* фасовка, указанная в накладной;  
* единица измерения из накладной;  
* количество по документу;  
* цена за единицу по документу;  
* сопоставленный товар из учетной системы;  
* учетная единица измерения из учетной системы.

4\. Этап 1\. Извлечение данных из накладной

На первом этапе OCR извлекает исходные данные из документа.

OCR не интерпретирует фасовку и не выполняет пересчет.

Результат OCR передается дальше как исходный текст и числовые значения, полученные из накладной.

5\. Этап 2\. AI-структурирование данных

AI-модуль анализирует распознанный текст и помогает выделить товарные позиции, наименование товара, возможную фасовку и единицу измерения.

AI на этом этапе не выполняет окончательный расчет.

Задача AI — подготовить данные для дальнейшей обработки системой.

6\. Этап 3\. Парсер фасовки

Парсер анализирует текст товара и поле фасовки.

Парсер должен определить:

* числовое значение фасовки;  
* единицу фасовки;  
* наличие упаковки;  
* возможную связь между количеством, упаковкой и учетной единицей.

Примеры:

* 250 г → 0,25 кг;  
* 500 мл → 0,5 л;  
* 1 кг → 1 кг;  
* 12 шт → 12 шт.;  
* 30 шт × 0,04 кг → 1,2 кг.

7\. Этап 4\. Стандартный арифметический пересчет

Если фасовка определена однозначно, система выполняет стандартный пересчет.

Примеры стандартного пересчета:

* граммы переводятся в килограммы;  
* миллилитры переводятся в литры;  
* количество упаковок умножается на вес или объем одной упаковки;  
* цена за упаковку пересчитывается в цену за учетную единицу.

На этом этапе Backend выполняет арифметику и формирует расчетные значения.

8\. Этап 5\. Проверка справочника исключений

Если стандартный пересчет невозможен или может дать ошибочный результат, система обращается к справочнику исключений.

Справочник исключений используется только для нестандартных случаев.

К таким случаям относятся:

* яйца с разным весом одной штуки;  
* штучные овощи и фрукты, учитываемые в килограммах;  
* товары с индивидуальным коэффициентом пересчета;  
* нестандартные упаковки;  
* позиции, где единица поставщика отличается от учетной единицы;  
* товары, для которых требуется специальное правило обработки.

Справочник исключений не заменяет автоматический анализ фасовок. Он используется только там, где стандартная логика не дает надежного результата.

9\. Этап 6\. AI-анализ спорных случаев

Если стандартные правила и справочник исключений не позволяют определить коэффициент пересчета, данные передаются в AI-модуль.

AI должен:

* проанализировать текст позиции;  
* определить возможную фасовку;  
* предложить предполагаемую единицу измерения;  
* предложить предполагаемый коэффициент пересчета;  
* вернуть степень уверенности;  
* указать необходимость проверки пользователем, если результат неоднозначен.

AI не должен формировать окончательное значение при низкой уверенности.

10\. Этап 7\. Ручное уточнение пользователем

Если система не может определить фасовку или коэффициент пересчета автоматически, строка передается пользователю на проверку.

Пользователь может уточнить:

* фасовку;  
* единицу измерения;  
* коэффициент пересчета;  
* сопоставленный товар.

После ручного уточнения система автоматически пересчитывает все зависимые расчетные поля.

11\. Этап 8\. Пересчет зависимых значений

После определения коэффициента пересчета система рассчитывает:

* количество в учетных единицах;  
* цену в учетных единицах;  
* сумму строки;  
* итоговые значения документа.

Все расчетные значения формируются системой.

Пользователь не должен вручную изменять расчетные поля напрямую. Исправления выполняются через исходные значения, от которых зависят расчеты.

12\. Этап 9\. Проверка результата

После пересчета система проверяет:

* определена ли единица измерения;  
* определена ли фасовка;  
* рассчитан ли коэффициент пересчета;  
* корректно ли рассчитано количество;  
* корректно ли рассчитана цена;  
* совпадает ли сумма строки с данными накладной;  
* есть ли ошибки или предупреждения.

Результат проверки передается на экран контроля качества данных.

13\. Итоговая логика

Архитектура обработки фасовок строится по принципу последовательного уточнения:

1. сначала используются данные накладной;  
2. затем выполняется парсинг фасовки;  
3. затем применяется стандартная арифметика;  
4. затем проверяется справочник исключений;  
5. затем применяется AI-анализ;  
6. при необходимости решение передается пользователю;  
7. после уточнения выполняется автоматический пересчет;  
8. перед экспортом результат проходит контрольную проверку.

14\. Архитектурное решение MVP

В рамках MVP не создается большой справочник всех возможных фасовок.

Основным решением является модуль автоматического анализа фасовок, работающий по цепочке:

парсер  
 ↓  
 арифметика  
 ↓  
 справочник исключений  
 ↓  
 AI  
 ↓  
 ручное уточнение пользователя

Такой подход снижает объем ручного ведения справочников, позволяет обрабатывать разные варианты записей поставщиков и сохраняет возможность контроля спорных случаев пользователем перед передачей данных в учетную систему.

# Анализ архитектуры обработки фасовок

**Документ для анализа архитектуры обработки фасовок**

**Проект: MVP продукта «АвтоСнаб»**

**1\. Цель документа**

Настоящий документ подготовлен для анализа предлагаемой архитектуры обработки фасовок в MVP продукта «АвтоСнаб».

Необходимо оценить техническую реализуемость подхода, при котором вместо большого справочника фасовок используется последовательная логика автоматической обработки:

парсер  
 ↓  
 арифметический пересчет  
 ↓  
 справочник исключений  
 ↓  
 AI / GPT-анализ  
 ↓  
 ручное уточнение пользователем при необходимости

**2\. Проблема**

В товарных накладных поставщики могут указывать фасовки и единицы измерения в разных форматах.

Примеры:

* 250 г;  
* 0,25 кг;  
* 500 мл;  
* 0,5 л;  
* 12 шт;  
* пачка 250 г;  
* упаковка 10 шт;  
* яйцо 30 шт;  
* товар в штуках, но учет ведется в килограммах.

Создание большого справочника всех возможных вариантов фасовок может привести к избыточному объему ручного ведения данных и не гарантирует покрытия всех вариантов записей поставщиков.

**3\. Предлагаемое решение**

Вместо большого справочника фасовок предлагается использовать модуль автоматического анализа фасовок.

Обработка должна выполняться последовательно:

1. Получение исходных данных из накладной.  
2. Распознавание текста OCR.  
3. Структурирование строки AI-модулем.  
4. Парсинг значения фасовки.  
5. Стандартный арифметический пересчет.  
6. Проверка справочника исключений.  
7. Передача спорных случаев в AI / GPT.  
8. Ручное уточнение пользователем, если автоматическое определение невозможно.  
9. Автоматический пересчет зависимых значений.  
10. Передача результата на экран контроля качества данных.

**4\. Что должен делать парсер**

Парсер должен попытаться определить из строки накладной:

* числовое значение фасовки;  
* единицу измерения фасовки;  
* количество упаковок;  
* связь между количеством, фасовкой и учетной единицей.

Примеры ожидаемой обработки:

* 250 г → 0,25 кг;  
* 500 мл → 0,5 л;  
* 1 кг → 1 кг;  
* 12 шт → 12 шт.;  
* 2 пачки по 250 г → 0,5 кг;  
* 30 шт × 0,04 кг → 1,2 кг;  
* 30 шт × 10 уп  ×  0,04 кг → 12,0 кг

**5\. Что должна делать арифметика**

Если фасовка определена однозначно, система должна выполнить стандартный пересчет.

Примеры:

* граммы → килограммы;  
* миллилитры → литры;  
* упаковки × вес упаковки;  
* упаковки × объем упаковки;  
* штуки × вес одной штуки при наличии правила;  
* цена за упаковку → цена за учетную единицу;  
* количество по документу → количество в учетных единицах.

**6\. Когда нужен справочник исключений**

Справочник исключений используется только для случаев, которые нельзя надежно обработать стандартной арифметикой.

Примеры:

* яйца с разным весом одной штуки;  
* консервированные продукты, имеющие сухой вес продукта;  
* штучные овощи и фрукты, учитываемые в килограммах;  
* товары с индивидуальным коэффициентом пересчета;  
* нестандартные упаковки;  
* позиции, где единица поставщика отличается от учетной единицы;  
* товары, для которых требуется специальное правило обработки.

Справочник исключений не должен превращаться в большой справочник всех фасовок.

Он должен хранить только правила для нестандартных случаев.

**7\. Когда подключается AI / GPT**

AI / GPT подключается только в тех случаях, когда:

* парсер не смог надежно определить фасовку;  
* арифметический пересчет невозможен;  
* в справочнике исключений нет подходящего правила;  
* строка содержит неоднозначное описание;  
* требуется интерпретация текста позиции.

AI / GPT должен вернуть:

* предполагаемую фасовку;  
* предполагаемую единицу измерения;  
* предполагаемый коэффициент пересчета;  
* степень уверенности;  
* признак необходимости ручной проверки.

Если уверенность недостаточна, результат не должен использоваться автоматически без проверки пользователя.

**8\. Роль пользователя**

Если система не смогла определить фасовку или коэффициент пересчета автоматически, строка должна быть передана пользователю на проверку.

Пользователь может уточнить:

* фасовку;  
* единицу измерения;  
* коэффициент пересчета;  
* сопоставленный товар.

После изменения данных система должна автоматически пересчитать:

* количество в учетных единицах;  
* цену в учетных единицах;  
* сумму строки;  
* итоговые значения документа.

Для проверки предлагаемой архитектуры нужно взять реальные строки из товарных накладных и протестировать несколько типов ситуаций.

* фрукты в штуках, учитываемые в килограммах;

Цель: определить, какие правила нужно хранить в справочнике исключений.

**9.4. Спорные случаи**

* сокращения;  
* отсутствующая единица измерения;  
* фасовка указана только в названии;  
* нестандартное написание;  
* несколько числовых значений в одной строке.

Цель: определить, в каких случаях нужен AI / GPT.

* что извлекает OCR;  
* что может определить парсер;  
* какой пересчет выполняется арифметикой;  
* требуется ли справочник исключений;  
* требуется ли AI / GPT;  
* требуется ли ручное уточнение пользователем.

**13\. Итоговая цель**

Итоговая цель анализа — подтвердить, что для MVP продукта «АвтоСнаб» можно отказаться от большого справочника фасовок и заменить его управляемой логикой:

парсер → арифметика → исключения → AI / GPT → ручная проверка

Такой подход должен снизить объем ручного ведения данных, ускорить обработку накладных и сохранить контроль качества перед загрузкой данных в учетную систему.

# Взаимосвязи столбцов

Эта таблица — не просто место, куда загружаются данные из фото накладной. Это промежуточный модуль проверки между первичным документом и учетной системой.  
В ней есть несколько разных типов колонок:

1. **Колонки из первичного документа**  
    Это то, что система распознала из фото/скана накладной:  
   * дата документа;  
   * номер документа;  
   * поставщик;  
   * наименование товара как написано в накладной;  
   * единица измерения как в накладной;  
   * количество как в накладной;  
   * цена;  
   * ставка НДС;  
   * сумма НДС;  
   * общая стоимость.

Эти данные нужны как “исходник”. Их нельзя сразу считать финальными для учетной системы, потому что в накладной товар и единицы могут быть написаны не так, как они заведены в УС.

2. **Колонки из справочников учетной системы**  
    После распознавания нужно сопоставить данные из накладной со справочниками УС:  
   * поставщик из справочника;  
   * товар из справочника УС;  
   * код/ID товара в УС;  
   * учетная единица;  
   * признаки ЕГАИС/Меркурий/Честный Знак, если они есть в карточке товара.

Поэтому в таблице есть два “наименования товара”:

* **наименование товара из документа** — как написано в накладной;  
  * **наименование товара в УС** — как этот товар должен называться в учетной системе.

Это не дублирование. Это связка “что пришло в документе” → “с чем сопоставили в справочнике”.

3. **Колонки пересчета фасовок и количества**  
    В накладной товар может быть указан в одном виде, а в учетной системе учитываться в другом.

Например:

* в документе: “Горчица дижонская, 250 г”, количество 20 банок;  
  * в УС нужно передать не “20 банок”, а количество в учетной единице, например в кг.

Поэтому нужны две разные колонки количества:

* **кол-во в документе** — что пришло из накладной;  
  * **кол-во в УС** — что получится после пересчета и пойдет в учетную систему.

Модуль фасовок должен вытащить из названия/описания товара объем или вес, определить фасовку, применить коэффициент и рассчитать финальное количество для УС.

4. **Где используется AI**  
    AI может использоваться там, где обычных правил недостаточно:  
   * разобрать неструктурированное название товара;  
   * понять, где в названии указана фасовка;  
   * помочь сопоставить товар из накладной с товаром из справочника;  
   * обработать нестандартные написания;  
   * предложить вариант, если точного совпадения нет.

Но AI не должен быть финальным источником истины. Он помогает предложить результат, а пользователь потом проверяет спорные строки.

5. **Справочник исключений**  
    Не все товары можно пересчитать простым правилом.

Например, яйца, штучные овощи, нестандартные упаковки или товары, где вес/объем нельзя надежно определить из названия. Для таких случаев нужен справочник исключений.  
Если строка требует исключения, система должна показать причину, а пользователь должен понимать, что эта позиция требует отдельной проверки.

6. **Колонки проверки и статусов**  
    После обработки строка получает статус:  
   * распознано;  
   * ошибка загрузки;  
   * исправлено вручную;  
   * отправлено в УС.

Также есть причина ручной корректировки:

* ошибка OCR;  
  * товар не найден в справочнике;  
  * требуется справочник исключений;  
  * ошибка сопоставления;  
  * другое.

Эти колонки не передаются в УС. Они нужны пользователю, чтобы понимать, какие строки готовы, а какие требуют исправления.

7. **Что отправляем в учетную систему**  
    В УС отправляем не всю таблицу, а только финальные подготовленные данные:  
   * дата документа;  
   * номер документа;  
   * поставщик;  
   * получатель/точка/склад;  
   * товар из справочника УС;  
   * учетная единица;  
   * количество в учетной единице;  
   * цена;  
   * ставка НДС;  
   * сумма НДС;  
   * итоговые суммы.

То есть в УС не отправляем “как было написано в накладной”, если это уже преобразовано. В УС должны уйти проверенные и сопоставленные данные.  
Главная логика такая:  
Фото накладной  
→ OCR распознал текст  
→ AI/парсер выделил строки и поля  
→ данные попали в колонки “из документа”  
→ товары и поставщики сопоставились со справочниками УС  
→ фасовки и единицы пересчитались  
→ пользователь проверил ошибки  
→ финальные данные ушли в УС  
Поэтому таблица содержит не только загруженные из фото поля, а весь путь данных: от исходного значения из накладной до финального значения, которое можно передать в учетную систему.  
Ключевой принцип:  
Колонка “из документа” показывает, что пришло.  
Колонка “из справочника/УС” показывает, с чем сопоставили.  
Колонка “расчет” показывает, что система пересчитала.  
Колонка “в УС” показывает, что реально пойдет в учетную систему.  
---

Если совсем коротко одной фразой:  
Эта таблица нужна не только для загрузки данных из фото, а для контроля полного преобразования: данные из накладной → сопоставление со справочниками → пересчет фасовок → проверка пользователем → финальная загрузка в УС.  
   
   
**Логика**  
**1\. Первичный документ**  
Фото, PDF, XML или другой файл накладной.  
   
**2\. Распознавание и разбор документа**  
Система считывает данные из документа.  
Для фото и сканов используется OCR.  
Для PDF, XML или Excel может использоваться парсер.  
AI может помогать разобрать сложные или нестандартные строки.  
   
**3\. Данные из документа**  
В таблицу попадает то, что было написано в накладной:  
дата, номер, поставщик, наименование товара, единица измерения, количество, цена, ставка НДС, суммы без НДС, с НДС.  
   
**4\. Сопоставление со справочниками учетной системы**  
Система сверяет данные из накладной со справочниками УС:  
\- поставщики (наименование, ИНН, КПП, код)  
\- товары (наименование, код, единицы измерения в учете, признаки Госсистемы: ЕГАИС / Меркурий / Честный Знак).  
 При сопоставлении привязка происходит строго по коду, а не по названию.  
   
**5\. Пересчет фасовок и единиц учета**  
Если в накладной товар в колонке количество указан в значении штучных единиц измерения и в названии товара указан в упаковках, штуках, бутылках, банках, килограммах, литрах и так далее, но явно в однозначной форме, то  
система пересчитывает это в учетную единицу, которая используется в УС.  
   
**6\. Если обычный пересчет невозможен**  
Строка отправляется на ручную проверку или обрабатывается через справочник исключений.  
   
**7\. Модуль проверки данных**  
Пользователь проверяет строки, ошибки и спорные позиции.  
Здесь видны статусы:  
Распознано,  
Ошибка загрузки,  
Исправлено вручную,  
Отправлено в УС.  
   
**8\. Финальные данные**  
После проверки остаются только подготовленные данные,  
которые можно передать в учетную систему.  
   
**9\. Учетная система**  
На основании этих данных создается документ поступления / приходная накладная.

 **Простыми словами:**  
Таблица — это не просто место, куда загружаются данные из фото.  
Это промежуточный модуль проверки между накладной и учетной системой.  
   
В таблице рядом находятся разные типы данных:  
1\. Что пришло из документа.  
2\. Что подтянули из справочников учетной системы.  
3\. Что рассчитала система.  
4\. Что пользователь проверил или исправил.  
5\. Что в итоге будет передано в учетную систему.  
   
   
 

| Группа колонок | Откуда берется | Для чего нужна |
| :---- | :---- | :---- |
| Данные из документа | OCR / парсер / AI из фото или файла накладной | Показать, что было в первичном документе |
| Наименование товара из документа | Из накладной | Сохранить исходное название, как написал поставщик |
| Товар в УС | Из справочника товаров учетной системы | Понять, какой именно товар должен попасть в учет |
| Кол-во в документе | Из накладной | Показать исходное количество поставщика |
| Кол-во в УС | Рассчитывается после фасовки и пересчета | Передать в учетную систему правильное учетное количество |
| Единица из документа | Из накладной | Показать, в чем поставщик указал товар |
| Единица в УС | Из справочника УС | Передать товар в той единице, в которой он ведется в учете |
| Фасовка / коэффициент | Парсер \+ AI \+ правила \+ справочник исключений | Пересчитать упаковки, банки, бутылки, кг, л и т.д. |
| Статус строки | Формируется системой и пользователем | Показать, готова строка или требует исправления |
| Причина корректировки | Система или пользователь | Объяснить, почему строка требует внимания |
| Финальные поля для УС | После проверки и пересчета | Используются для создания документа поступления |

   
   
   
**Финальные поля для УС** — это не все колонки таблицы, а только те данные, которые нужны учетной системе для создания документа поступления.  
   
В нашем тестовом варианте в УС должны уходить такие поля:  
   
**1\. Дата документа**  
Дата накладной.  
   
**2\. Номер документа**  
Номер накладной поставщика.  
   
**3\. Поставщик**  
Поставщик, сопоставленный со справочником учетной системы.  
   
**4\. ИНН поставщика**  
Нужен для идентификации поставщика и проверки сопоставления.  
   
**5\. Получатель**  
Кто получает товар.  
   
**6\. Торговая точка**  
Объект / заведение / подразделение, куда поступает товар.  
   
**7\. Склад**  
Склад или место хранения, куда приходуется товар.  
   
**8\. Основание**  
Например, договор, заказ или другое основание поступления, если это требуется в УС.  
   
**9\. Наименование товара в УС**  
Не название из накладной, а товар, сопоставленный со справочником учетной системы.  
   
**10\. Ставка НДС**  
Например 10% или 20% / 22%, в зависимости от данных документа и правил учета.  
   
**11\. Сумма НДС**  
Сумма налога по строке.  
   
**12\. Общая стоимость**  
Итоговая стоимость строки.  
   
**13\. Сумма накладной**  
Общая сумма документа, если она нужна для сверки и загрузки.  
   
**14\. Единица измерения в УС**  
Та единица, в которой товар ведется в учетной системе.  
   
**15\. Количество в УС**  
Не обязательно количество из накладной, а итоговое количество после пересчета фасовки.  
   
**16\. Цена в УС**  
Цена, рассчитанная или приведенная к учетной единице.  
   
**То есть логика такая:**  
   
из накладной мы берем исходные данные,  
из справочника УС берем правильный товар, поставщика и единицы учета,  
через модуль фасовки пересчитываем количество,  
после проверки оставляем финальный набор полей,  
и именно этот набор передаем в УС для создания документа поступления.  
   
**В УС не должны уходить служебные колонки проверки**: готовность к загрузке, статус строки, причина ручной корректировки. Эти поля нужны только внутри модуля проверки, чтобы пользователь видел, что исправлено, что готово, а что требует внимания.  
   
**Что Нужно Понять**  
**Наименование товара из документа**  
→ нужно для проверки, что пришло от поставщика.  
**Наименование товара в УС**  
→ нужно для загрузки в учетную систему.  
Это два разных поля, потому что поставщик может написать товар как угодно, а в учетной системе он должен быть привязан к конкретной карточке товара по его коду.  
   
**Количество из документа**  
→ показывает, что написано в накладной.  
**Количество в УС**  
→ показывает, сколько нужно передать в учетную систему после пересчета фасовки.  
Это тоже два разных поля, потому что в накладной может быть “20 банок по 250 г”, а в учетную систему нужно передать, например, 5,000 кг.  
   
**Короткая Формула**  
Из документа берем исходные данные.  
Из учетной системы берем справочники.  
Система сопоставляет товар и поставщика.  
Модуль фасовок пересчитывает количество.  
Пользователь проверяет ошибки.  
В УС отправляем только финальные проверенные данные.  
   
   
   
**Еще короче**  
Колонки “из документа” нужны, чтобы видеть исходник.  
Колонки “из справочника УС” нужны, чтобы понимать, с чем сопоставили.  
Колонки “расчет” нужны, чтобы пересчитать фасовки и количество.  
Колонки “статус/причина” нужны для проверки и исправлений.  
В учетную систему уходят только финальные проверенные поля, а не вся таблица.

**Вариант 1\. Ручной Для MVP**  
 Самый простой сценарий, чтобы быстро проверить бизнес-логику:

1\. Накладная пришла в ЭДО: СБИС / Астрал / другой оператор.

2\. Пользователь скачивает файл документа.

3\. Если есть XML — лучше скачивать XML.

4\. Если XML нет, скачивает PDF.

5\. Файл загружается в наш сервис / таблицу.

6\. Система разбирает данные.

7\. Данные попадают в модуль проверки.

8\. Пользователь проверяет.

9\. Готовые данные передаются в УС.

Для MVP это нормальный путь, потому что он быстрее для проверки гипотезы.

 

**Вариант 2\. Автоматический Через Интеграцию**  
 сценарий:

1\. Документ поступает в ЭДО.

2\. Наш сервис получает документ автоматически через API ЭДО.

3\. Забирает XML / PDF / вложения.

4\. Разбирает документ.

5\. Передает данные в модуль проверки.

6\. Пользователь проверяет только спорные строки.

7\. После подтверждения данные уходят в учетную систему.

То есть пользователь уже не скачивает файл вручную.

 

**Что Лучше Забирать Из ЭДО**  
 Если есть выбор, лучше всего:

XML → самый удобный вариант

PDF → хуже, если это скан или картинка

Фото/скан → нужен OCR

Excel/CSV → тоже удобно, если структура стабильная

XML хорош тем, что там данные уже структурированы: поставщик, покупатель, товары, количество, цена, НДС, суммы. Их не надо “читать глазами” через OCR.

 

**Как Это Может Быть Технически**  
 Возможные способы:

API ЭДО

Интеграционный шлюз

Выгрузка в папку / облачное хранилище

Передача через email

*Обмен через учетную систему, если ЭДО уже связан с iiko – в данном случае Нет*

Ручная выгрузка файлов на этапе MVP

 

**Что Нужно Уточнить У Интегратора ЭДО**  
 Вот список вопросов:

1\. Есть ли API для получения входящих документов?

2\. Можно ли получать документы автоматически?

3\. В каком формате доступны документы: XML, PDF, ZIP, JSON?

4\. Можно ли получать только накладные / УПД / счета-фактуры?

5\. Можно ли получать статус документа: новый, подписан, отклонен, обработан?

6\. Можно ли получать вложения документа?

7\. Нужен ли API-ключ, токен, сертификат или ЭЦП?

8\. Можно ли настроить уведомление о новом документе?

9\. Есть ли тестовый контур?

10\. Кто со стороны ЭДО/клиента может дать доступ и документацию?

 

**Логика**

На этапе MVP допускается ручная загрузка файлов из ЭДО в модуль проверки данных. Приоритетный формат — XML, так как он содержит структурированные данные документа и снижает зависимость от OCR.

 

В целевой версии необходимо предусмотреть автоматическое получение документов из ЭДО через API или интеграционный шлюз. После получения документ разбирается, данные приводятся к единой структуре, проходят сопоставление со справочниками, пересчет фасовок и попадают в модуль проверки перед передачей в учетную систему.

# **Методика внедрения MVP**

Методика внедрения MVP продукта «АвтоСнаб» определяет порядок перехода от ручной обработки товарных накладных к автоматизированной обработке данных.  
 Главная задача внедрения — проверить работу продукта на реальных документах без нарушения текущих бизнес-процессов предприятия.  
Внедрение выполняется поэтапно. Каждый следующий этап начинается после проверки стабильности предыдущего.

## **12.1. Подготовительный этап**

На подготовительном этапе анализируется текущий процесс обработки накладных: как документы поступают, кто их проверяет, какие данные вводятся вручную, какие справочники используются и как выполняется загрузка в учетную систему.  
Также подготавливаются справочные данные:

* справочник товаров;  
* справочник поставщиков;  
* единицы измерения;  
* данные по складам, торговым точкам и организациям;  
* справочник исключений для нестандартных фасовок, если он используется.

Отдельно проверяется техническая готовность: загрузка документов, распознавание данных, сопоставление со справочниками, пересчет фасовок, работа модуля проверки данных и формирование данных для передачи в учетную систему.  
Пользователи знакомятся с порядком работы: загрузка документа, проверка строк, исправление ошибок, запуск проверки и подготовка данных к передаче в учетную систему.

## **12.2. Пилотный запуск**

Пилотный запуск проводится на ограниченном количестве реальных накладных и пользователей.  
 Цель этапа — проверить, как MVP работает на практических данных.  
Для каждой накладной проверяется полный цикл:

* загрузка документа;  
* распознавание данных;  
* сопоставление товаров и поставщиков;  
* пересчет фасовок и единиц учета;  
* проверка пользователем;  
* исправление ошибок;  
* подготовка данных к загрузке в учетную систему.

По итогам обработки фиксируются ошибки, предупреждения, спорные позиции и случаи ручной корректировки. Критические проблемы устраняются до расширения использования MVP.

## **12.3. Параллельный режим работы**

После пилотного запуска MVP используется параллельно с действующим процессом учета.  
 Одна и та же накладная обрабатывается двумя способами:

* по текущему ручному или действующему процессу предприятия;  
* через MVP.

Результаты сравниваются между собой. Проверяются товарные позиции, сопоставление со справочниками, единицы измерения, количество, цены, суммы, НДС и готовность данных к загрузке в учетную систему.  
Все расхождения классифицируются по причинам: ошибка распознавания, отсутствие данных в справочнике, ошибка сопоставления, нестандартная фасовка, ошибка пересчета или ручная корректировка пользователя.

## **12.4. Анализ результатов и корректировка**

После пилота и параллельной проверки анализируются результаты работы MVP:

* качество распознавания документов;  
* корректность сопоставления товаров и поставщиков;  
* правильность пересчета фасовок;  
* количество ручных исправлений;  
* причины ошибок;  
* удобство работы пользователей;  
* готовность данных к передаче в учетную систему.

По итогам анализа формируется список доработок. Могут уточняться правила сопоставления, параметры обработки фасовок, справочник исключений, контрольные проверки, формат выгрузки и пользовательский интерфейс.  
После изменений проводится повторная проверка затронутых этапов обработки.

## **12.5. Расширение использования MVP**

После подтверждения стабильной работы MVP область применения постепенно расширяется.  
 Допускается увеличение количества накладных, подключение новых пользователей, поставщиков, заведений и разных типов документов.  
При расширении контролируются:

* скорость обработки документов;  
* количество ошибок;  
* количество ручных корректировок;  
* корректность сопоставления;  
* успешность подготовки данных к загрузке в учетную систему.

Если качество обработки ухудшается, расширение временно ограничивается до устранения причин.

## **12.6. Переход к рабочей эксплуатации**

Переход к рабочей эксплуатации возможен после подтверждения стабильной работы MVP на реальных данных.  
К моменту перехода должны быть выполнены условия:

* полный цикл обработки накладной работает без критических ошибок;  
* данные корректно проходят проверку перед загрузкой;  
* результаты обработки соответствуют требованиям учетной системы;  
* пользователи понимают порядок работы с модулем;  
* спорные и ошибочные строки корректно выводятся на ручную проверку.

После перехода к рабочей эксплуатации MVP используется как основной инструмент обработки товарных накладных в рамках реализованного функционала.

## **12.7. Критерии успешного внедрения**

Внедрение MVP считается успешным, если:

* система обрабатывает товарную накладную от загрузки до подготовки данных к учетной системе;  
* данные проходят проверку перед передачей;  
* ошибки и спорные позиции выявляются до загрузки;  
* пользователи могут самостоятельно работать с модулем;  
* количество ручных операций сокращается;  
* результаты обработки можно использовать для создания документа поступления в учетной системе;  
* архитектура подтверждает возможность дальнейшего масштабирования.

### **Итог внедрения MVP**

Внедрение MVP строится как контролируемый переход от текущего процесса обработки накладных к автоматизированной обработке. Такой подход позволяет проверить продукт на реальных данных, снизить риски внедрения и определить необходимые доработки до расширения системы.

# Обработка статусов и корректировок

**Обработка статусов и корректировок в модуле проверки данных**  
   
В модуль проверки данных загружаются данные из первичного документа: фото, PDF, XML, Excel или другого файла накладной. Один документ может содержать несколько товарных строк.  
   
Статусы документа заполняются только в первой строке документа. Это относится к колонкам **«Статус загрузки»**, **«Статус строки»**, **«Дубль»**, а также к основным реквизитам документа: дата, номер, поставщик и другие поля уровня документа.  
   
Колонка **«Корректировка»** заполняется точечно в той товарной строке, где выявлена проблема.  
   
**1\. После загрузки данных в таблицу**  
Если документ распознан без замечаний:  
Статус загрузки \= Проверить  
Статус строки \= Распознано  
Корректировка \= пусто  
   
Если найдена проблема, которую можно проверить и исправить вручную:  
Корректировка \= Нет в справочнике / Исключение / Сопоставление / Другое  
Статус загрузки \= Требует проверки  
Статус строки \= Правка вручную  
   
Если выявлена ошибка OCR:  
Корректировка \= Ошибка OCR  
Статус загрузки \= Не готово  
Статус строки \= Ошибка загрузки  
   
Ошибка OCR означает, что документ плохо распознан. В этом случае проблему нужно решать на этапе загрузки/распознавания: загрузить более качественный файл, PDF/XML вместо фото или повторить распознавание.  
   
Если документ определен как дубль:  
Дубль \= Да  
Статус загрузки \= Не готово  
Статус строки \= Распознано  
Если документ похож на дубль, но нужно решение пользователя:  
Дубль \= ?  
Статус загрузки \= Требует проверки  
Статус строки \= Распознано  
   
**2\. Действия пользователя при проверке**  
Пользователь смотрит строки, где заполнена колонка **«Корректировка»**.  
   
Если причина:  
Нет в справочнике  
Исключение  
Сопоставление  
Другое  
пользователь исправляет данные: выбирает товар, сопоставляет позицию, применяет исключение, корректирует значение.  
   
После исправления пользователь меняет в первой строке документа:  
Статус строки \= Распознано  
Статус загрузки руками менять не нужно. Его пересчитает скрипт после проверки.  
   
Если в колонке **«Дубль»** стоит ?, пользователь должен принять решение:  
если это дубль:  
\- в колонке «Дубль» поставить Да;  
\- документ останется в статусе Не готово и не будет загружаться.  
если это не дубль:  
\- очистить ячейку «Дубль»;  
\- в первой строке документа поставить Статус строки \= Распознано;  
\- при необходимости поставить Статус загрузки \= Проверить;  
\- поставить галочку в колонке «Загрузка»;  
\- нажать «Проверить выбранные документы».  
   
**3\. Проверка выбранного документа**  
   
Пользователь ставит галочку в колонке **«Загрузка»** в первой строке нужного документа и нажимает:  
Проверка / Загрузка → Проверить выбранные документы  
   
Если проблем больше нет:  
Статус загрузки \= Загрузить  
Статус строки \= Готов к загрузке  
Корректировка \= очищается  
   
Если остались причины корректировки:  
Статус загрузки \= Требует проверки  
Статус строки \= Правка вручную  
   
Если осталась ошибка OCR или подтвержденный дубль:  
Статус загрузки \= Не готово  
   
**4\. Загрузка в тестовый лист / учетную систему**  
   
После успешной проверки пользователь снова ставит галочку у документа и нажимает:  
Проверка / Загрузка → Загрузить на тестовый лист  
   
Загружаются только документы со статусом:  
Статус загрузки \= Загрузить  
Статус строки \= Готов к загрузке  
   
После загрузки:  
Статус загрузки \= Загружено  
Статус строки \= Отправлено в УС  
   
Галочка в колонке **«Загрузка»** снимается автоматически.  
   
Документы со статусами **«Не готово»** и **«Требует проверки»** не загружаются. Галочки с них тоже снимаются после попытки загрузки.  
   
**5\. Возврат загруженного документа на проверку**  
   
Если после загрузки выяснилось, что документ нужно исправить, пользователь ставит галочку в первой строке документа и нажимает:  
Проверка / Загрузка → Вернуть на проверку  
   
После этого:  
Статус загрузки \= Требует проверки  
Статус строки \= Возврат на проверку  
   
Пользователь вносит исправления. После исправления он ставит:  
Статус строки \= Распознано  
   
Затем снова выполняет обычную последовательность:  
Проверить выбранные документы  
→ если все хорошо, Загрузить на тестовый лист  
   
**6\. ВАЖНО**  
Названия колонок в строке 2 менять нельзя без согласования, потому что скрипт ищет колонки по названиям.  
   
Строки 1 и 2 защищены:  
 строка 1 содержит пояснения для разработчиков, строка 2 содержит рабочие названия колонок.  
   
Статусы документа должны заполняться только в первой строке документа. Не нужно проставлять **«Статус загрузки»** и **«Статус строки»** в каждой товарной строке.  
   
Причины корректировки должны заполняться в конкретной товарной строке, где выявлена проблема.  
   
При загрузке нескольких документов подряд нужно визуально разделять документы между собой: например, добавлять нижнюю границу у последней строки документа или пустую строку-разделитель. Это нужно, чтобы пользователь видел границы документов и таблица не воспринималась сплошным полотном.

# **1\. Зачем нужен этот документ**

Этот документ объясняет, как устроен модуль проверки данных в MVP АвтоСнаб и как через него должна проходить товарная накладная. Он написан простым языком для команды, разработчиков, куратора и пользователей, чтобы всем было понятно, какие данные откуда берутся, почему в таблице есть несколько похожих на первый взгляд колонок и какие действия выполняет пользователь перед загрузкой в УС (iiko, SH ).

Под УС дальше понимается учетная система ресторана или сети заведений, например iiko или SH (Store House). В текущем MVP реальная передача в учетную систему заменена тестовой загрузкой на отдельный лист, но логика подготовки данных уже построена так, как она должна работать перед реальной интеграцией.

# **2\. Общая последовательность работы**

Маршрут документа выглядит так: пользователь или система получает первичный документ, данные из него считываются, затем попадают в модуль проверки, после этого сопоставляются со справочниками учетной системы, пересчитываются по фасовкам и учетным единицам, проверяются пользователем и только после этого готовятся к загрузке в учетную систему.

| Шаг | Что происходит | Результат |
| ----- | ----- | ----- |
| 1 | Поступает первичный документ: фото, скан, PDF, XML, Excel или файл из ЭДО. | Есть исходный файл накладной. |
| 2 | Система считывает данные. Для фото и сканов используется OCR, для структурированных файлов \- парсер. | Получены строки и реквизиты документа. |
| 3 | Данные попадают в модуль проверки. | Пользователь видит, что система прочитала из документа. |
| 4 | Поставщик и товары сопоставляются со справочниками учетной системы. | Появляются внутренние товары, единицы учета, коды и признаки госсистем. |
| 5 | Модуль фасовок определяет объем, вес, упаковку и пересчитывает количество. | Формируется количество и цена в учетных единицах. |
| 6 | Пользователь проверяет спорные места и исправляет ошибки. | Документ получает статус готовности к загрузке. |
| 7 | Готовые данные передаются в учетную систему или в тестовый лист. | В УС можно создать документ поступления. |

# **4\. Что заполняется при загрузке документа**

При загрузке документа в таблицу сначала должны попасть данные, которые пришли из самой накладной. Это дата и номер документа, поставщик, ИНН поставщика, получатель, торговая точка, склад, основание, наименование товара из документа, единица измерения из документа, количество по документу, цена, ставка НДС, сумма НДС, общая стоимость и сумма накладной.

Важно разделять данные уровня документа и данные уровня товарной строки. Дата документа, номер документа, поставщик, ИНН, форма документа и статус документа относятся ко всей накладной. Поэтому они заполняются только в первой строке документа. Наименования товаров, количество, цена и причины корректировки относятся к конкретным товарным позициям, поэтому они могут заполняться построчно.

| Почему реквизиты документа стоят только в первой строке Одна накладная может содержать много товаров. Если в каждой товарной строке повторять дату, номер и поставщика, таблица станет визуально тяжелой и неудобной. Поэтому реквизиты документа показываются один раз \- в первой строке этого документа, а товарные позиции идут ниже. |

| Реквизиты документа | Форма документа, Дата документа, № Документа, Поставщик, ИНН Поставщика, Грузоотправитель, Получатель, Торговая точка, Склад, Основание | Это данные накладной и заведения. Они нужны для идентификации документа и будущего создания поступления. |
| Данные из накладной | Наименование товара из документа, Ед.изм. в документе, Кол-во в документе, Цена за ед\-цу, Стоимость без НДС, Ставка НДС, Сумма НДС, Общая стоимость, Сумма накладной | Это то, что система считала из первичного документа. Пользователь сверяет эти значения с оригиналом. |
| Данные из справочников УС | Поставщик, товар в УС, Ед.изм. в УС, Госсистемы, внутренние коды поставщиков и товаров | Нужны, чтобы в учетную систему ушли не свободные названия из накладной, а правильные учетные объекты. |
| Расчетные поля | Кол-во в УС, Цена в УС, суммы и пересчитанные значения | Формируются после сопоставления товара и пересчета фасовки в учетную единицу. |
| Контроль снабжения | Кол-во в заявке, Цена по прайсу, Предыдущая дата поставки, Предыдущая цена, Отклонение от цены прайса | Помогают проверить, выполнил ли поставщик условия заявки, прайса и договора. |
| Служебные поля | Время загрузки документа, ID документа, Ссылка на исходный документ | Нужны для технической трассировки: когда загружен документ, какой у него ID и где лежит исходный файл. |

# **6\. Почему есть два наименования товара и два количества**

В таблице есть наименование товара из документа и наименование товара в УС. Это не дублирование, а две разные сущности. Наименование из документа показывает, как товар написал поставщик в накладной. Наименование в УС показывает, к какому внутреннему товару учетной системы эту строку нужно привязать.

Поставщик может написать товар по-своему: сокращенно, с ошибками, с фасовкой внутри названия или с коммерческим названием. В учетной системе товар должен быть создан и загружен по внутреннему справочнику, желательно по коду товара, а не по свободному текстовому названию. Поэтому модуль сначала показывает исходное название из накладной, а потом результат сопоставления со справочником товаров.

Такая же логика у количества. Кол-во в документе \- это количество, как его указал поставщик. Кол-во в УС \- это количество после пересчета в учетную единицу. Например, поставщик указал 24 бутылки по 0,5 л, а учет ведется в литрах. Тогда в документе будет 24 бутылки, а в учетную систему должно уйти 12 литров.

# **7\. Сопоставление поставщиков и товаров**

После загрузки данных система должна сопоставить поставщика из накладной со справочником поставщиков учетной системы. Для надежной работы итоговая привязка должна идти по внутреннему коду или идентификатору поставщика в УС. Наименование и ИНН помогают найти нужную карточку, но финально в учетную систему должен передаваться именно учетный объект, а не просто текст из накладной.

Затем система сопоставляет каждую товарную строку со справочником товаров. Здесь также важно, что название используется для поиска и проверки, но правильная привязка должна быть к товару из справочника УС по его названию коду. Из карточки товара подтягивается учетное наименование, единица учета и признаки госсистем, например ЕГАИС, Меркурий или Честный Знак, если такие признаки ведутся в карточке.

Если товар или поставщик не найден, строка не должна молча уходить дальше. Она должна получить причину корректировки: Нет в справочнике. Если товар найден, но есть несколько похожих вариантов и нужно выбрать правильный, используется причина Сопоставление. Если позиция требует специального правила пересчета, используется причина Исключение.

# **8\. Фасовки, искусственный интеллект и пересчет**

Самая важная часть модуля \- пересчет фасовок и единиц измерения. Поставщик может указать товар в упаковках, бутылках, банках, штуках, килограммах или литрах. Учетная система при этом может вести тот же товар в другой единице. Задача модуля \- понять, как перевести количество из документа в количество, которое нужно передать в УС.

Сначала система берет исходное наименование товара, единицу измерения, количество и цену из накладной. Затем модуль анализа фасовок пытается выделить из названия вес, объем, количество в упаковке или другой признак фасовки. Для простых случаев достаточно парсера и арифметики: граммы переводятся в килограммы, миллилитры в литры, упаковки умножаются на вес или объем одной упаковки.

Искусственный интеллект нужен не вместо всех расчетов, а как помощник в сложных строках. Он может помочь разобрать нестандартное название, выделить предполагаемую фасовку, единицу и коэффициент пересчета. Но если уверенность низкая или строка неоднозначная, результат должен идти пользователю на проверку, а не автоматически загружаться в учетную систему.

| Ситуация | Как обрабатывается | Что видит пользователь |
| ----- | ----- | ----- |
| 250 г, 500 мл, 1 кг | Обычный парсер и арифметический пересчет. | Строка может пройти без ручной корректировки. |
| Упаковка 10 шт, бутылка 0,5 л | Количество умножается на фасовку или объем упаковки. | В Кол-во в УС появляется пересчитанное значение. |
| Яйцо, овощи поштучно, нестандартная банка | Может потребоваться справочник исключений. | В Корректировка появляется Исключение. |
| Неоднозначное название или несколько чисел | AI-модуль предлагает трактовку, при сомнении строка идет на проверку. | Документ получает Требует проверки. |
| Плохое распознавание текста | Это проблема OCR, а не фасовок. | Документ получает Не готово и Ошибка загрузки. |

# **9\. Расчет цен, НДС и сумм**

После определения учетной единицы система рассчитывает количество в УС и цену в УС. Если поставщик указал цену за упаковку, а учет ведется в килограммах или литрах, цена должна быть пересчитана в учетную единицу. Затем проверяются суммы: стоимость без НДС, ставка НДС, сумма НДС, общая стоимость и сумма накладной.

НДС и суммы нужны не только для красивого отображения. Они помогают проверить, что строка после пересчета не потеряла смысл: количество, цена и итоговая сумма должны сходиться с накладной. Для учетной системы обычно передаются ставка НДС, сумма НДС, общая стоимость, сумма накладной, количество и цена в учетной единице.

Стоимость без НДС может использоваться как контрольное поле. Если учетная система сама пересчитывает стоимость без НДС на основании общей стоимости и ставки НДС, это поле может не входить в минимальную выгрузку. Но для проверки внутри модуля оно полезно, потому что помогает быстрее увидеть расхождение.

# **10\. Статусы и причины корректировки**

Статусы нужны, чтобы пользователь и скрипт понимали, что делать с документом дальше. Важно: Статус загрузки и Статус строки относятся к документу целиком и заполняются только в первой строке документа. Корректировка относится к конкретной товарной строке и может быть заполнена там, где находится проблема.

| Поле | Значение | Что означает |
| ----- | ----- | ----- |
| Статус загрузки | Проверить | Документ загружен в модуль и ожидает проверки пользователем или скриптом. |
| Статус загрузки | Требует проверки | Есть вопрос, который можно разобрать вручную: сопоставление, исключение, нет в справочнике или другое. |
| Статус загрузки | Не готово | Документ нельзя загружать: есть Ошибка OCR или подтвержденный дубль. |
| Статус загрузки | Загрузить | Документ прошел проверку и готов к тестовой или реальной загрузке. |
| Статус загрузки | Загружено | Документ уже отправлен дальше в УС. |
| Статус строки | Распознано | Документ или строка прочитаны без ошибки распознавания. |
| Статус строки | Правка вручную | Пользователь должен проверить и исправить данные. |
| Статус строки | Ошибка загрузки | Проблема на этапе распознавания или загрузки документа. |
| Статус строки | Готов к загрузке | Проверка пройдена, документ можно загружать. |
| Статус строки | Отправлено в УС | Данные переданы дальше в УС. |
| Статус строки | Возврат на проверку | Ранее загруженный документ вернули для исправления. |

# **11\. Что означает колонка Корректировка**

Колонка Корректировка показывает причину, почему товарная строка требует внимания. Она не передается в учетную систему, потому что это служебная информация для пользователя и команды. После успешной проверки документа причины корректировки очищаются, чтобы пользователь не видел старую историю как новую проблему.

| Причина | Когда появляется | Что делает пользователь |
| ----- | ----- | ----- |
| Нет в справочнике | Товар или поставщик не найден в справочнике УС. | Добавляет позицию в справочник или выбирает корректную карточку. |
| Сопоставление | Есть несколько похожих вариантов или нужно подтвердить правильную связь. | Выбирает правильный товар или поставщика. |
| Исключение | Обычный пересчет фасовки не подходит. | Применяет правило из справочника исключений или уточняет коэффициент. |
| Другое | Проблема есть, но она не попадает в стандартные причины. | Разбирает ситуацию вручную. |
| Ошибка OCR | Документ плохо считан. | Повторяет загрузку/распознавание или использует более качественный файл. |

# **12\. Как пользователь проверяет выбранные документы**

Пользователь выбирает документ галочкой в колонке Загрузка. Галочка ставится только в первой строке документа. Если документ состоит из десяти товарных строк, галочка все равно нужна только в первой строке. Скрипт воспринимает эту галочку как выбор всего документа.

После этого пользователь нажимает в верхнем меню: Проверка / Загрузка \-\> Проверить выбранные документы. Скрипт проверяет только выбранные документы, а не всю таблицу. Если в документе нет дубля, нет ошибки OCR и все причины корректировки устранены, документ получает Статус загрузки \= Загрузить, а Статус строки \= Готов к загрузке. Причины корректировки по этому документу очищаются.

Если в документе остались причины корректировки, он остается в статусе Требует проверки. Если есть Ошибка OCR или Дубль \= Да, документ получает Не готово. Если в колонке Дубль стоит знак вопроса, пользователь должен принять решение: очистить знак вопроса, если это не дубль, или заменить его на Да, если это дубль.

# **13\. Как выполняется загрузка в учетную систему**

В текущем MVP реальная интеграция с УС (iiko, SH ) пока заменена листом Загрузка тест. Это сделано безопасно: мы можем проверить, какие строки подготовлены к передаче, какие поля попадают в выгрузку и как меняются статусы, не затрагивая настоящую учетную базу.

Для загрузки пользователь ставит галочки у документов, которые нужно отправить, и нажимает: Проверка / Загрузка \-\> Загрузить на тестовый лист. Скрипт загружает только те документы, у которых Статус загрузки \= Загрузить и Статус строки \= Готов к загрузке. Документы со статусами Не готово и Требует проверки не загружаются.

После успешной загрузки в исходной таблице меняются статусы: Статус загрузки \= Загружено, Статус строки \= Отправлено в УС. Галочки в колонке Загрузка снимаются автоматически и у загруженных, и у пропущенных документов. Это защищает пользователя от случайной повторной загрузки.

# **14\. Какие поля передаются в учетную систему**

В УС (iiko, SH ) передается не вся таблица. Передаются только те поля, которые нужны для создания документа поступления или приходной накладной. Служебные поля, статусы и причины корректировки остаются в модуле проверки.

| Передаваемое поле | Зачем нужно в УС |
| ----- | ----- |
| Дата документа | Дата первичной накладной. |
| № Документа | Номер накладной поставщика. |
| Поставщик | Контрагент документа поступления. |
| ИНН Поставщика | Контроль идентификации поставщика. |
| Получатель | Кто получает товар. |
| Торговая точка | Объект сети, к которому относится поступление. |
| Склад | Склад или место хранения. |
| Основание | Договор, заказ или другое основание. |
| Наименование товара в УС | Сопоставленный товар из справочника учетной системы. |
| Ставка НДС | Налоговая ставка для строки. |
| Сумма НДС | Сумма налога по строке. |
| Общая стоимость | Стоимость строки с учетом НДС. |
| Сумма накладной | Итоговая сумма документа. |
| Ед.изм. в УС | Учетная единица измерения. |
| Кол-во в УС | Количество после пересчета в учетную единицу. |
| Цена в УС | Цена в учетной единице. |

# **15\. Какие поля не передаются в учетную систему**

В учетную систему не передаются Статус загрузки, Статус строки, Корректировка, Дубль, Загрузка, Время загрузки документа, ID документа, ссылка на исходный документ и поясняющие служебные поля. Эти колонки нужны внутри модуля проверки: они помогают пользователю понять, что исправить, а разработчикам \- отследить маршрут обработки.

Также в учетную систему обычно не нужно передавать исходное наименование товара из документа, если уже найден правильный товар в УС. Исходное название остается в модуле для сверки с накладной и для разбора спорных случаев.

# **16\. Возврат документа на исправление**

Иногда документ уже загрузили, а потом выяснилось, что поставщик прислал исправление или в накладной была ошибка по цене, количеству или товару. Для таких случаев нужен контролируемый возврат документа на проверку.

Пользователь ставит галочку в первой строке загруженного документа и нажимает: Проверка / Загрузка \-\> Вернуть на проверку. После этого документ получает Статус загрузки \= Требует проверки и Статус строки \= Возврат на проверку. Пользователь вносит исправления, затем в первой строке документа ставит Статус строки \= Распознано и снова запускает Проверить выбранные документы. Если все хорошо, документ снова получает Загрузить и может быть отправлен дальше.

| Что важно при возврате Не нужно вручную менять каждую товарную строку большого документа. Управляющие статусы стоят в первой строке документа. Корректировки по товарам заполняются только там, где действительно есть проблема. |

# **17\. Правила для разработчиков**

Ниже правила, которые важно соблюдать при дальнейшей разработке, чтобы не ломать текущую логику таблицы и скриптов.

| Правило | Почему это важно |
| ----- | ----- |
| Не менять названия колонок в строке 2 без согласования. | Скрипт ищет столбцы по названиям. Переименование может сломать проверку и загрузку. |
| Строки 1 и 2 должны быть защищены от случайного редактирования. | Строка 1 объясняет происхождение данных, строка 2 содержит рабочие названия колонок. |
| Статус загрузки и Статус строки заполнять только в первой строке документа. | Это статусы документа целиком, а не каждой товарной позиции. |
| Корректировку заполнять в конкретной товарной строке. | Пользователь должен видеть, какая именно позиция требует внимания. |
| Галочка Загрузка ставится только в первой строке документа. | Она означает выбор всего документа для проверки или загрузки. |
| Документы нужно визуально разделять. | Без разделения несколько накладных превращаются в сплошное полотно и пользователь теряет границы документов. |
| Для визуального разделения лучше использовать нижнюю границу последней строки документа. | Пустая строка может мешать обработке данных, а граница не ломает структуру. |
| Ошибка OCR не должна исправляться бизнес-скриптом статусов. | Это проблема качества распознавания. Нужно повторное распознавание или более качественный источник. |
| Справочник исключений не должен превращаться в большой справочник всех фасовок. | Он нужен только для нестандартных случаев, где обычная арифметика ненадежна. |

# **18\. Быстрый сценарий для пользователя**

В обычной работе пользователь действует так. Сначала он загружает документ или получает уже загруженные строки в модуле проверки. Затем смотрит документы со статусами Требует проверки и Не готово. Если причина корректировки относится к справочнику, сопоставлению, исключению или другому ручному вопросу, пользователь исправляет данные и ставит в первой строке документа Статус строки \= Распознано.

После исправления пользователь ставит галочку в колонке Загрузка на первой строке документа и запускает Проверить выбранные документы. Если документ прошел проверку, он получает Статус загрузки \= Загрузить и Статус строки \= Готов к загрузке. После этого пользователь снова выбирает документ галочкой и нажимает Загрузить на тестовый лист или, в будущем, загрузить в учетную систему.

Если документ загружен, но позже его нужно исправить, пользователь выбирает его галочкой и запускает Вернуть на проверку. После внесения исправлений документ снова проходит тот же маршрут: Проверить выбранные документы \-\> Загрузить.
